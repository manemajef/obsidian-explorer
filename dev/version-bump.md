# Version Bump And Release Guide

This is the local checklist for preparing an Explorer release. The normal
working branch is `dev`; releases are made from `main`. Keep `dev` at least as
new as `main`.

## Preflight

- Read [warnings.md](./warnings.md) before bumping. Do not reintroduce warnings
  that Obsidian has already complained about.
- Check the worktree with `git status --short --ignored data.json main.js`.
  `data.json` and `main.js` are local/ignored and must not drive release
  decisions.
- Build before and after the bump with `npm run build`.
- Release from committed source, not from local plugin state.
- Review commits and diffs since the last release tag before writing release
  notes.

Useful commands:

```bash
git describe --tags --abbrev=0
git log --oneline "$(git describe --tags --abbrev=0)..HEAD"
git diff --stat "$(git describe --tags --abbrev=0)..HEAD"
git diff --name-only "$(git describe --tags --abbrev=0)..HEAD"
```

## Version Choice

- Unless explicitly requested otherwise, bump the patch version:
  `1.x.y -> 1.x.(y + 1)`.
- Patch numbers can pass 9. For example, `1.1.9 -> 1.1.10`.
- Keep the tag exactly equal to `manifest.json.version`. The release workflow
  fails if they differ.

## Files To Update

Update these together:

- `manifest.json`: `"version"`
- `package.json`: `"version"`
- `package-lock.json`: generated by npm after `package.json` changes
- `versions.json`: add `"new.version": "minAppVersion"`
- `changelog.md`: add a top release entry
- `styles.css`: generated by `npm run build` when CSS changes

Do not update:

- `data.json`: ignored local plugin settings, not shipped
- `main.js`: ignored local build output; GitHub Actions builds the release asset

## Changelog Style

Keep release notes user-facing and feature-focused. Avoid implementation-heavy
phrasing unless the change is mostly maintenance.

Use this shape:

```md
## YYYY-MM-DD - Version X.Y.Z

**Release:** short user-facing summary

**Highlights:**
- ...
- ...
- ...
```

If the main work is internal refactoring, describe the benefit plainly, such as
“settings reliability improvements” or “general performance improvement.”

## Bump Workflow

From `dev`:

1. Review commits and diffs since the last tag.
2. Update version files and changelog.
3. Run `npm install --package-lock-only` if `package-lock.json` did not update
   after changing `package.json`.
4. Run `npm run build`.
5. Inspect `git diff`. Confirm ignored files are not part of the release.
6. Commit with a clear message, usually `Prepare X.Y.Z release`.

Then release from `main`:

```bash
git checkout main
git pull
git merge dev
git push origin main
git tag X.Y.Z
git push origin X.Y.Z
```

The tag push runs `.github/workflows/release.yml`, which validates that the tag
matches `manifest.json`, installs dependencies, builds, attests provenance for
`manifest.json`, `main.js`, and `styles.css`, and creates/uploads the GitHub
release assets.

## Post-Release Checks

- Confirm the GitHub release exists for the tag.
- Confirm assets are attached: `manifest.json`, `main.js`, `styles.css`.
- Confirm the release workflow completed successfully and build provenance
  attestation was generated.
- Return to `dev` and make sure it is not behind `main`.
